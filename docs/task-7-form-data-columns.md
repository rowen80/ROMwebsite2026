## Goal
Add a small, intentional block of new columns to `2026 FORM_DATA` so the rebuilt form can preserve scheduling/logistics data without disturbing the familiar core workflow.

## Guiding principle
Keep the current core columns intact and familiar.
Add new rebuild-only scheduling/logistics fields to the **right side** of the existing sheet.

This preserves:
- copy/paste familiarity
- current operational rhythm
- existing invoice-oriented columns
- easier fallback comparison with the old workflow

## Why these columns

### Listing/logistics details
- `BuildingName`
- `Bathrooms`
- `Views`
- `FinishedBasement`

These preserve property details that may matter for scheduling, expectations, and future website scheduling features.

### Timing/scheduling details
- `DateListingReady`
- `DateToGoLive`
- `DesiredDate`
- `IsVacant`
- `DuringShootAgreement`

These are the main future scheduling and operational-planning fields.

### Access details
- `AccessType`
- `AccessCode`
- `OwnerContactInfo`

These matter operationally, but do not belong in invoice-delivery messaging.

### Notes / traceability
- `PhotographerNotes`
- `TermsAccepted`
- `SourceSystem`
- `RequestId`

These preserve operational context and make debugging much easier.

### Automation visibility
- `ConfirmationEmailStatus`
- `InternalEmailStatus`

These are optional but recommended. They let you see, in the sheet, whether the rebuilt automation sent the expected messages.

## Recommended value formats

### `Views`
Store as a comma-separated string.
Example:
`Ocean, Bay, Golf Course`

### `DuringShootAgreement`
Store as:
- `TRUE` / `FALSE`
or
- `Yes` / `No`

Recommendation: use `TRUE` / `FALSE` consistently.

### `TermsAccepted`
Store as:
- `TRUE`

This should always be true for successful submissions.

### `SourceSystem`
Use a fixed identifier:
- `romwebsite2026`

### `RequestId`
Use a unique id generated by Render/backend.
Examples:
- UUID
- timestamp + short random suffix

### Email status columns
Use simple status values:
- `PENDING`
- `SENT`
- `FAILED`

These can be written by n8n.

## Recommended non-changes
Do **not** change these current columns for V1:
- `Estimated Line Items`
- `Estimated Total`
- `Total`
- `BillingEmail`
- `Message`
- invoice/delivery columns

## Suggested placement strategy
Append all new columns to the far right of the current used range.
Do not insert them in the middle of the existing intake columns.

Why:
- lower disruption
- easier rollback
- fewer accidental formula/layout side effects
- preserves current visual memory
